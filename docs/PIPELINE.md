# CI/CD Pipeline — целевой дизайн

> Документ описывает **дизайн, к которому мы стремимся**.
> Источник — визуальная схема на доске (стики и текст) и договорённости начального дизайна.
> Не путать с фактическим состоянием кода — здесь зафиксирована цель.

## Схема

```
[1] pre-commit            локально, до каждого коммита
        │                 ruff, тесты, секреты — агент не коммитит при падении
        ▼
[2] push → feature/*      Feature CI: static + security параллельно
        │                 быстрый сигнал, без деплоя
        ▼
[3] PR открыт             полный прогон: tests + AI-review комментами в PR
        │                 человек смотрит findings, апрувит
        ▼
[4] merge → main          собираются только изменённые образы → деплой на dev
        │
        ▼
[5] release tag (v*)      сборка всех образов → деплой на prod
```

Отдельно: **ручной запуск** (`workflow_dispatch`) — собрать образ(ы) и при желании
задеплоить на выбранный стенд, вне основного потока.

---

## 1. pre-commit

**Где:** локально, на машине разработчика / агента.
**Когда:** перед каждым `git commit`.
**Цель:** поймать очевидное до того, как оно уйдёт в remote.

Шаги:
- `ruff check --fix` — линт с автоисправлением
- `ruff format` — форматирование
- `gitmark lint` — KB валидна (I1–I6) *(временно отключено)*
- `trufflehog` — скан секретов

**При падении любого шага — коммита не происходит.**

---

## 2. push → feature CI

**Где:** CI (GitHub Actions).
**Когда:** каждый push в `feature/**`.
**Цель:** быстрый сигнал (< 3 мин), fail-fast. Без тяжёлых тестов и без деплоя.

Три job'а параллельно:
- **static** — `ruff` (check + format) на весь проект
- **security** — `semgrep OWASP`, `pip-audit`, `trufflehog`
- **tests** — только лёгкие: `pytest -m "not heavy"` (heavy-тесты сюда не попадают)

Сборка образов на feature-push **не запускается**. Образы собираются:
- автоматически — при merge в `main` (только изменённые, см. этап 4)
- вручную — через `workflow_dispatch` (перебилд любого образа на любой стенд)

Умная пересборка: `detect-changes` (paths-filter) + registry cache.

---

## 3. PR: tests + AI-review

**Где:** CI (GitHub Actions).
**Когда:** открытие PR и каждый push в ветку PR.
**Цель:** полная валидация перед merge.

PR — это **gate перед merge**, а не старт пайпа. К моменту открытия PR
static и security уже прошли на push (этап 2). Человек смотрит на зелёный CI +
комментарии ревьюера, апрувит.

На этапе PR работают **три механизма**:

### 3.1. tests

Полный прогон в отдельных job'ах — **включая heavy** (в отличие от feature push,
где они пропускались):
- `pytest tests/unit`
- `pytest tests/integration`

Тесты помечаются маркером `@pytest.mark.heavy` (медленные/дорогие: e2e, load).
На push в feature гоняется `pytest -m "not heavy"` — быстрый сигнал; на PR
прогоняется всё. `codex-review` ждёт прохождения всех тест-job'ов (`needs`),
чтобы не ревьюить заведомо сломанный код.

### 3.2. Codex review — авто на открытии + по команде

Codex-ревью — **часть CI**, а не забываемое ручное действие, но и не на каждый
коммит (иначе жжём квоту подписки):

- **автоматически при открытии PR** (`opened`), **после прохождения тестов** —
  `job codex-review` в `pr.yml`, `needs: [unit-tests, integration-tests]`,
  `if: github.event.action == 'opened'`;
- **по команде `@codex review`** в комментарии PR — свежее ревью после правок,
  когда человек его явно хочет (`codex-command.yml`);
- тесты при этом гоняются на **каждый** push (а Codex — нет).

Как работает сам ревьюер (общий скрипт `.github/scripts/codex_review.py`):
- `runs-on: [self-hosted, codex]`, авторизация — **подписка ChatGPT**
  (`auth.json` на раннере), не OpenAI API (см. 3.4);
- берёт diff PR, гоняет `codex exec` → **структурированный JSON** находок;
- постит **одну GitHub-review с inline-тредами** на конкретные строки +
  summary с вердиктом (LGTM / needs changes); что не легло на строку — в summary;
- escape-hatch: label `skip:ai-review` отключает авто-ревью на открытии.

`concurrency` в `pr.yml` отменяет устаревшие прогоны при быстрых push'ах.

Это «первый проход» — машина отсеивает очевидное до того, как PR смотрит человек.

### 3.3. @claude — по требованию (Anthropic)

Любой член команды может прямо в PR написать комментарий с упоминанием
`@claude` (например, *«@claude почини то, что нашёл Codex»*) — и Claude
внесёт правки сам.

- workflow на событии `issue_comment` / `pull_request_review_comment`
- условие: тело комментария содержит `@claude`
- используется официальный `anthropics/claude-code-action`
- Claude читает контекст PR + текст обращения, вносит изменения,
  **пушит коммит в ветку PR** и отвечает в треде

Разделение ролей: **Codex находит** (автоматически, на каждый PR),
**Claude чинит** (по явному запросу человека).

```
PR открыт ──▶ tests + Codex review (авто) ──▶ человек смотрит findings
                                                      │
                                       «@claude почини X» в комментарии
                                                      ▼
                                   Claude правит → коммит в ветку PR → повтор CI
```

### 3.4. Как это устроено: self-hosted runner на подписке

Codex-ревью из 3.2 крутится на подписке ChatGPT, а не на OpenAI API.
`openai/codex-action` для этого не подходит — он работает только на API-ключе
(`OPENAI_API_KEY`), подписочный `auth.json` он не принимает. Поэтому вызываем
`codex exec` напрямую на self-hosted runner'е, где сохранён логин Codex.

Принцип:
- на runner'е лежит `~/.codex/auth.json` (вход подпиской) → расход из плана;
- job `codex-review` (`runs-on: [self-hosted, codex]`) запускается **автоматически**
  после тестов (см. 3.2) и зовёт `codex exec`, постит ревью в PR.

**Разовая настройка runner'а** (на доверенной машине):
1. `codex login` → создаёт `~/.codex/auth.json`. Проверить, что внутри
   `"auth_mode": "chatgpt"` и есть refresh-токен (при необходимости
   `cli_auth_credentials_store = "file"`).
2. Зарегистрировать self-hosted runner **на репозиторий**:
   Settings → Actions → Runners → New self-hosted runner
   (`./config.sh --url <repo> --token <reg-token>`, затем `./run.sh` или сервис).

**Обращение с `auth.json` в CI (критично):**
- класть из секрета **только если файла нет** — иначе затрёшь обновлённые токены;
- после прогона **сохранить обновлённый** `auth.json` (Codex рефрешит токен, если
  `last_refresh` старше ~8 дней);
- относиться к нему как к паролю (не коммитить, не шарить);
- не использовать один `auth.json` в параллельных джобах / на нескольких машинах →
  один runner, задачи последовательно.

**Ограничения:**
- только **приватные** репо (self-hosted + публичный = форки исполняют код на твоей
  машине);
- квота — из подписки (общая, лимиты плана);
- runner надо держать живым (аптайм, обновления Codex CLI);
- OpenAI помечает путь как «advanced» и для тяжёлого CI рекомендует API-ключ.

**Масштаб:** на личном аккаунте runner регистрируется per-repo. Чтобы один runner
обслуживал много репо — нужен **org-level runner** (требует GitHub Organization).

### Ключи (секреты) — что, где, зачем

| Секрет / креды | Кто использует | Откуда берётся |
|---|---|---|
| `GITHUB_TOKEN` | оба — для постинга комментов и пуша | **автоматически** выдаётся Actions, заводить не нужно (нужен только блок `permissions:` в job) |
| `~/.codex/auth.json` (на runner'е) | Codex review | `codex login` на self-hosted runner'е — НЕ секрет репо, см. 3.4 |
| `CLAUDE_CODE_OAUTH_TOKEN` | `@claude` fixer | `claude setup-token` → кладёшь в Settings → Secrets |

Codex не использует секрет репо вообще — авторизация лежит на самом раннере
(`auth.json`). Для `@claude` токен лежит зашифрованным в секретах и пробрасывается
как `claude_code_oauth_token: ${{ secrets.CLAUDE_CODE_OAUTH_TOKEN }}`; в логах
GitHub маскирует значения секретов.

`GITHUB_TOKEN` — особенный: его не нужно создавать, Actions генерирует его на
каждый запуск. Чтобы он мог писать комменты и пушить, в job задаётся:

```yaml
permissions:
  pull-requests: write
  contents: write
```

---

## 4. merge → main → dev

**Когда:** push/merge в `main`.
**Цель:** автодеплой на dev-стенд.

- `detect-changes` (paths-filter) определяет затронутые сервисы
- собираются и пушатся в GHCR **только изменённые образы**, тег = short SHA
- деплой **только изменённых** сервисов на **dev** (порт api 8080)

---

## 5. release tag → prod

**Когда:** проставление релиз-тега `v*`.
**Цель:** деплой на прод.

- сборка **всех** образов с тегом релиза (`:v0.1.0` + `:stable`)
- деплой **всех** сервисов на **prod** (порт api 8090)

---

## Ручной запуск (workflow_dispatch)

Вне основного потока — кнопка «Run workflow» в GitHub UI:

- **service** — какие сервисы собирать (список через запятую или `all`)
- **deploy** — деплоить ли после сборки
- **environment** — целевой стенд (любой, что заведён в Settings → Environments;
  валидируется через GitHub API перед сборкой)
- **branch** — выбирается встроенным селектором веток

---

## Стенды и секреты

- Стенды живут в **Settings → Environments** (dev, prod, и любые другие).
- На каждый стенд — секреты `SSH_HOST`, `SSH_USER`, `SSH_KEY`.
- Список стендов нигде не захардкожен — добавляешь environment в Settings,
  ручной воркфлоу деплоит туда без правок кода.

### Механизм деплоя

Вся логика — в `scripts/deploy.sh` (не bash-простыня в yaml). Каждый deploy-job:

1. `actions/checkout` → `appleboy/scp-action` копирует `docker-compose.yml` +
   `scripts/deploy.sh` в `/srv/deploy/<env>/` на сервере;
2. `appleboy/ssh-action` запускает `deploy.sh <env> <owner/repo> <tag> [services]`.

`deploy.sh`:
- `docker login ghcr.io` кредами из `GHCR_USER`/`GHCR_TOKEN` (= `GITHUB_TOKEN`) —
  образы приватные;
- изоляция стендов на одном хосте: `COMPOSE_PROJECT_NAME=afc_<env>` + свой
  `API_PORT` (dev 8080, prod 8090);
- `docker compose pull/up -d` для нужных сервисов (пусто/`all` = все).

**Имена образов:** build и compose используют один префикс
`ghcr.io/<owner/repo>/<service>` — иначе compose не найдёт собранный образ.

> Прим.: в текущем демо один сервер эмулирует и dev, и prod (разные порты/проекты).
> В реале — разные хосты, адреса в секретах соответствующих environment.
