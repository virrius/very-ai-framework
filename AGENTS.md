# AGENTS.md — правила для кодового агента

Эти правила обязательны при разработке в репозитории с этим CI/CD.
Цель — чтобы изменения проходили пайплайн с первого раза и безопасно доезжали до прод.

## Рабочий цикл

1. **Ветка** — всегда от `main`, имя `feature/<кратко-суть>`. В `main` напрямую не коммитим.
2. **Перед каждым коммитом — pre-commit** (он же финальный шаг разработки):
   - `ruff format` + `ruff check --fix`
   - `pytest` (юнит)
   - скан секретов (trufflehog)
   - **любой шаг упал → НЕ коммитим**, чинимся.
3. **push** в `feature/*` → Feature CI (static ‖ security ‖ light tests). Чинь красное сразу.
4. **PR в `main`** → PR CI: unit + integration + автоматический Codex review.
   - Разбери находки Codex. Нужен свежий проход — коммент `@codex review`.
   - Хочешь, чтобы правки внёс агент — коммент `@claude fix <что>`.
   - **Не мёржим красный CI.** Merge — только зелёный + аппрув человека.
5. Merge в `main` → автодеплой на **dev**. Релиз — тег `vX.Y.Z` → **prod**.

## Правила кода и тестов

- **Не выходи за рамки задачи** — объём изменений не должен заметно превышать запланированное.
- **Сервис = каталог `services/<имя>/`** со своим `Dockerfile`. Новый сервис — просто
  новый каталог; пайплайн подхватит его сам (имена нигде не зашиты).
- **Тесты:** быстрые — в `tests/unit`, тяжёлые/медленные (e2e, load) помечай
  `@pytest.mark.heavy`. На push в feature гоняются только лёгкие (`-m "not heavy"`),
  на PR — все.
- **ruff запинен** (`pyproject.toml` + `.pre-commit-config.yaml` + CI — одна версия).
  Меняешь версию — синхронно во всех трёх местах, иначе CI и локаль разойдутся.
- **Линию длины и правила ruff** — из `pyproject.toml`, не переопределяй локально.

## Секреты и конфиг

- **Никогда** не коммить ключи/токены/`.env` с секретами в репозиторий.
- Несекретный конфиг → **Environment Variables**: `SSH_HOST`, `SSH_USER`
  (видно, какой сервер к какому стенду), вся `.env` в `APP_DOTENV`.
- Секреты → **Environment Secrets**: `SSH_KEY`, `APP_SECRET`, … (всё, что даёт доступ).
- Принцип: секрет — только то, что **открывает доступ**; адреса/имена — это конфиг
  (Variables). Помни: Variables видны в логах, Secrets маскируются.
- Новый параметр приложения — добавь строку в `APP_DOTENV` нужного окружения (через UI),
  без правок workflow. Новый секрет — Environment Secret + одна строка в шаге сборки `app.env`.

## Триггеры (что когда запускается)

| Действие | Что происходит |
|---|---|
| `git commit` (локально) | pre-commit: формат, lint, тесты, секреты |
| push в `feature/**` | Feature CI: static ‖ security ‖ light tests |
| открытие PR в `main` | PR CI: unit + integration + авто Codex review |
| коммент `@codex review` | повторный Codex review |
| коммент `@claude ...` | Claude вносит правки и пушит в ветку PR |
| merge в `main` | build изменённых образов → deploy **dev** |
| тег `v*` | build всех образов → deploy **prod** |
| Actions → Run workflow (Manual) | ручной build + опционально deploy на выбранный стенд |

## Что НЕ делает пайплайн (намеренно)

- На push в feature — **не** собирает образы и **не** деплоит (только быстрые проверки).
- Codex review на PR — **только при открытии** (не на каждый push), чтобы не жечь квоту;
  повторно — по `@codex review`.
- Порты/реплики/конфиг приложения пайплайн не навязывает — это дело `docker-compose.yml`
  и Environments конкретного проекта.

Подробный разбор каждого этапа — в `docs/PIPELINE.md`.
